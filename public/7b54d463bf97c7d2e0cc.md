## Kinesisは「流れ続けるデータ」をそのまま扱う仕組み 🌊

Kinesisは、  
**今この瞬間も増え続けているデータ**を前提にしたサービス。

ログ、クリックイベント、IoTデータなど、  
止めてから処理する余裕がないデータを、そのまま受け止める。

いわゆる **ETL（Extract / Transform / Load）** を  
リアルタイム寄りにやるための仕組み。

## ETLって何をする？ 🔄

ETLはざっくり言うと、

- データを集めて（Extract）
- 使いやすく整えて（Transform）
- 保存・分析先に流す（Load）

という流れ。

AWSだと、  
Data Pipeline、Glue、そして **Kinesis** がこの役割を担う。

## Kinesisファミリーの役割分担 📦

Kinesisは一枚岩じゃない。  
用途ごとにサービスが分かれている。

### Data Streams
- シャード管理あり
- 低レイテンシ
- 細かく制御したいリアルタイム処理向け

### Data Firehose
- シャード管理なし
- 設定がシンプル
- S3やRedshiftにそのまま流したい時に便利

### Video Streams
- 映像データ用
- 監視カメラや動画解析向け

### Data Analytics
- 収集したデータをその場で分析・可視化
- ストリーミングSQL的な立ち位置

## サーバレスETL代表：Glue 🧩

ETLといえば、もう一人の常連が **Glue**。

Glueは、
データウェアハウスやデータレイク前提のサーバレスETL。

流れはだいたいこんな感じ。

1. データをクロール  
2. メタデータを Data Catalog に保存  
3. 抽出・変換して保存先へ  
   （Redshift、RDS、S3、オンプレなど）

S3 → Redshift みたいな  
「貯めてから整える」処理が得意。

## ざっくり使い分けメモ 📝

- **Kinesis**  
  → 流れてくるデータを今すぐ扱う
- **Glue**  
  → 溜まったデータを整理して渡す

同じETLでも、  
時間軸がまるで違う。

AWSのデータ系サービス、  
「いつ処理するか」で性格が分かれるのが面白い。
